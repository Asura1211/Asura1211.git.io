11/13 Node.js

Node.js 简介

  1. Node.js不是一门编程语言，而是一个运行时
  2. 编写程序的语言是JavaScript
  3. Node.js靠Chrome V8引擎运行JavaScript

Node.js 模块系统 NPM

NPM 是为了让 Node.js 的文件可以相互调用的一个简单的模块管理系统。

模块机制：

模块加载过程流程图：

Node.js 文件模块的应用

  • Grunt：任务驱动的构建工具，有大量现成的插件封装了常见的任务，能管理任务之间的依赖关系，自动化地执行依赖的任务。
  • Glup：基于流的自动化构建工具。以管理和执行任务，还支持监听文件、读写文件。
  • Webpack：打包模块化JavaScript的工具。在Webpack里一切文件皆模块，通过Loader转换文件，通过Plugin注入钩子，最后输出由多个模块组合成的文件。

如何提高Node服务器的开发效率

Node只提供了基础的web服务器能力。但企业级框架（express、koa、egg）将路由、逻辑、视图、服务、任务等分离，结构清晰，职责分明，学习和维护成本低。

  • Express：Node界最老牌的Web框架。提供了基础的请求上下文封装（封装比较齐全），路由，中间件，模板引擎接口，静态资源服务。
  • Koa：Express原班人马打造，定义为下一代Web开发框架。仅提供一个请求上下文，中间件机制，没有捆绑任何中间件（便于二次开发）。通过利用async/await，让你写出更优雅的代码，方便开发者针对自身需求开发自定义框架。
  • Egg.js ：阿里推出的一款web企业级框架。提供了完整的Web开发所需的组件，使用 koa 作为底层。使用了约定大于配置的理念提供强大的插件机制，方便扩展。

向egg服务器发起一次请求，背生了什么

  • Egg服务器收到请求
  • URL路径匹配Router.js的路由
  • 执行路由指向的controller
  • Egg-Nunjucks模版引擎生成html
  • Egg服务器返回html代码

以后有机会再跟着钉钉群回放敲一遍！

前端工程本质上还是解决如何提升效率和质量，降低成本。（多快好省）

工程体系概述

前端工程发展历程：

构建与调试

  • 构建的目标是如何更好的治理资源
  • 调试的目标是如何更好的提升开发效率

babel

Babel 转译器，将最新的JS语言规范转换成浏览器可支持的代码版本，而不局限于浏览器是否支持，Babel类I具极大的促进了JS语言规范的繁荣。

webpack

Webpack 静态模块打包工具，require everything，bundle everything，所以也叫file bundler，loader 和 plugins 机制，促使它一骑绝尘，奠定今天前端构建工具领域的主导地位。（loader代替了grunt任务运行，plugins代替了require等）

Webpack loader 与 plugins：

loader：是一个转换器，将A文件进行编译成B文件。比如：将A.less转换为A.css， 单纯的文件转换过程（file级别）

  1. 通过css-loader 提取JS文件中的样式编译为 css文件
  2. 通过csso-loader对css进行优化
  3. 最后合并到一个style.css文件

plugins：是个扩展器，它丰富了webpack本身，针对是loader结束后，webpack 打包的整个过程，它并不直接操作文件，而是基于事件机制工作，会监听webpack打包过程中的某些节点，执行广泛的任务。（bundle chunk级别）

  1. 通过css-loader 提取JS文件中的样式编译为 css 文件
  2. 通过csso-webpack plugin对css进行压缩，体积减小50%
  3. 最后合并到一个style.css文件

Dev Server

开发服务器，在本地开发调试前端应用，修改代码后能自动编译加载并刷新浏览器，展示新的内容。

node本地服务——监听文件变化——编译改动后的文件——刷新页面展示

HMR

Hot Module Replacement 模块热替换

  1. 不会重新加载整个页面，仅替换变更的模块，可以保留页面状态
  2. 增量更新，节约开发时间，更快展示界面

单测与持续集成

  • 单测和持续集成，解决的是产品质量，多人协同的问题。

单元测试：用来对一个模块、一个函数或者一个类来进行正确性检测的测试工作。

断言(assert)：用来确定某段程序执行结果应该是某个值的预期。

代码覆盖率：四个测量维度

  1. 行覆盖率：每一行
  2. 函数覆盖率：每个函数
  3. 分支覆盖率：每个代码块
  4. 语句覆盖率：每个语句

Cl 持续集成（Continuous Integration）：通过自动化测试等方法，将代码频繁的集成到主干，以达到快速迭代的同时，还能保持高质量。优势：

  1. 快速发现错误
  2. 防止分支大幅偏离主干

监控

  • 监控的目标是更好的保障产品线上平稳运行及更好的服务用户。

异常监控：执行环境监控

指标监控：衡量工程化能力的监控

埋点监控：业务监控

监控过程：

  • 采集：异常数据、性能数据、用户行为数据
  • 上报：延时、去重、压缩、上报
  • 分析
  • 报警

    面向面试：基础一定要好（3+浏览器DOM）、找练手项目（能够解决实际问题）

  9. （公司）软件开发流程：

  10. 如何看待 lowcode：

  12. 前端智能化：

昭朗——ECMAScript与语言标准规范

（有点没太听懂）

什么是ECMAScript

Web脚本语言：为了能让开发者通过脚本语言动态操作网页元素，JavaScript 作为Web脚本语言诞生，并通过Web衍生了庞大而富饶的JavaScript技术生态。

Node.js/ Serverless：ECMAScript作为一种基于对象(Object)和事件驱动(Event Driven)的轻量级编程语言，与服务端、Serverless 场景的异步编程模型也非常契合。

谁负责定义ECMAScript

ECMAScript标准化组织

Ecma International：国际标准化组织，负责如ECMAScript, C#等标准定义

TC39：全权负责ECMAScript的标准ECMA262的编撰

ECMAScript会有什么变化

ECMAScript 2015: Promise、Class

ECMAScript 2017: Async Functions

ECMAScript 2018: Promise.finally

ECMAScript 2022: Top Level Await、Class字段、Class私有字段

“我”能做什么

  • 通过书籍、导引学习ECMAScript,阅读ecma262不是必要选项
  • 学习过程中对于ECMAScript有任何问题，可以通过MDN => ecma262的路径自己排查问题（带着问题阅读ecma262 ）
  • 通过https://github.com/tc39/proposals深入各种提案，了 解未来ECMAScript可能的设计
  • 加入开源项目!(如Node.js, V8， Chromium 等等)

豆酱——如何参与开源项目

一个目标

  1. 是什么？组件?工具?框架?
  2. 为什么？当前的问题是什么?我们为什么可以解决它?
  3. 怎么做？内部产品开源?内外共建的开源?（最流行的方式，构建一个自运行的社区）个人维护的开源项目?

一套规范

  1. 项目规范：每个人都会有自己的风格，从lint规范到开发风格各不相同。没有统一的项目规范会使得代码间过于跳跃，甚至难以理解。
    • 代码规范。i++ ++i 等
    • 格式化规范。
    • 命名结构规范（API naming standard）
  2. 轮值规范：靠爱发电很宝贵，但是没有规范约束的爱会随着时间消散。一套轮值规范帮助你将项目长远的持续下去。Issue、PR处理、Feature讨论、版本发布(PR 合并、日志)、站点发布。
  3. 贡献规范：打破圈子，通过社区的力量将你的项目越做越好。建立一套贡献规范，让志愿者可以快速的参与进来。代码分支、开发流程、PR细则。

一个社区

  1. Issue交流：维护你的 issue 列表，能够向你开 issue 的用户弥足珍贵。注意Issue与Discussion的区别。
  2. PR处理：（To Contributor）从一个Bug/Feature开始，按照贡献规范搭建环境并开发，不要惧怕交流。
  3. 更多核心贡献者：长期参与的开发者会逐渐成为核心开发者。不用担心自身技术的深浅，因为你的代码无论多少都可以惠及更多的开发者。（定期的参与 lssue、PR，投票进入核心贡献者群体）
